# 下一步路线建议

## 1. 当前优先级最高的事项

### P0：历史页继续收紧并定稿

历史页已经从扫描页分离出来，但还没有完全收口。当前最优先的是把它稳定成真正的 archive / recent processed list：

- 修正左侧 `番号` 列的对齐锚点
- 调整历史页行高、列宽、留白
- 继续弱化操作感
- 确认状态 badge 尺寸和时间列密度

### P0：批处理反馈继续打磨

当前 batch rail 已经具备雏形，但还需要继续稳定：

- 完善终态文案
- 进一步优化折叠态
- 明确失败项在列表中的高亮与筛选方式
- 设计“只看失败项”的合理入口

### P0：失败重试能力

当前失败项可以留在列表中，但还缺一个明确的“再试一次”路径。建议下一步补上：

- 对失败项单独重试
- 或只重试当前批次失败项

## 2. 建议的实现顺序

建议按下面顺序继续：

1. 历史页布局收口
2. 批处理状态条与失败态精修
3. 失败重试能力
4. 刮削能力的数据结构预留
5. 刮削 UI 入口设计
6. 元数据能力落地

这个顺序的原因是：

- 先把现有整理闭环打稳
- 再基于稳定的记录页和批任务体系接刮削

## 3. v1 必做 / 可后置

### v1 必做

- 历史页最终定稿
- 批处理失败反馈更清楚
- 失败重试
- stale selection / stale status 场景继续打稳
- `test_data` 与 UI smoke 流程的可重复性

### 可以后置

- 更复杂的历史检索
- 多维度高级筛选
- 批任务持久化到数据库
- 批任务历史列表
- 更复杂的目录打开 / 文件系统联动

## 4. 刮削能力预留建议

用户已经明确提到，后续页面会加入刮削功能，因此现在就应该避免把现有结构写死。

### 数据层预留

建议未来在 `files` 或独立元数据表中补充：

- `scrape_status`
- `scraped_at`
- `metadata_source`
- `nfo_generated`
- `poster_generated`

### UI 层预留

- 历史页可以继续保留“记录列表”定位，但需要预留刮削状态位
- 状态 badge 系统未来应支持：
  - 已处理
  - 刮削中
  - 已刮削
  - 刮削失败

### 交互层预留

- 批处理 rail 的思想可以复用于未来刮削任务
- 不建议单独做一个完全不同风格的“刮削任务中心”

## 5. 与当前实现最相关的收口建议

### 历史页

- 用一轮专门的视觉 QA 把历史页与扫描页的差异固定下来
- 历史页要更像日志，不像确认页

### 批处理

- 运行中自动展开已合理
- 终态保留结果直到手动收起更合理
- 下一步关键不是加更多按钮，而是让失败路径更可继续

### 复制反馈

- 当前局部 toast 已经比顶部全局提示合理
- 等刮削和更多 hover 动作进入后，再统一复查一次位置、时长和层级

## 6. 文档与配置建议

在继续开发前，建议后续顺手做两件事：

1. 把旧的 `phase1-*` 文档和当前 README 做一次同步检查
2. 继续坚持“环境差异只进 profile / env，不写死进代码”

## 7. 一句总结

当前最合理的下一步不是继续堆新按钮，而是：

先把“扫描页闭环 + 历史页收口 + 批处理失败路径”打稳，再在这个稳定骨架上接入刮削能力。
