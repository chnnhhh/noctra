# UI / 交互规范

## 1. 当前已确定的风格关键词

Noctra 当前 UI 已经明确收敛到这条方向：

- dark
- console
- devtool
- terminal
- cold color
- restrained glow
- precise, not playful

可以参考的气质关键词：

- GitHub Dark
- VS Code
- Raycast
- Terminal UI

明确不是：

- 传统 SaaS admin
- 白底后台表格
- 大色块运营面板
- 过强浮层感的 dashboard

## 2. 页面定位差异

### 扫描页

定位是“确认工作台”：

- 信息密度更高
- 强调输入与输出的对照
- 支持筛选、分页、排序、跨页选择
- 强调待处理项的批量操作

### 历史页

定位是“整理记录 / archive list”：

- 更紧凑
- 更弱操作感
- 更适合快速回看最近处理结果
- 应明显区别于扫描页，不应直接复用扫描页的大表格节奏

## 3. 表格布局原则

### 扫描页

- 左侧是输入：`番号 / Code`、`源文件 / Source`
- 右侧是系统输出：`整理结果 / Output`
- 状态列需要给 integrated expanding badge 预留空间
- 行 hover 使用整行冷色光晕，不按单元格切开

### 历史页

- 更紧凑
- `源文件` 收成 1 行
- `整理结果` 最多 2 行
- `状态` 是记录态 badge，不应像可点击按钮
- `整理时间` 保留两行日期/时间

## 4. 状态 badge 语义色

### 扫描结果

- 待处理：蓝色系
- 已存在：橙色系
- 已处理：绿色系
- 未识别：克制的偏红系
- 失败：低饱和红

### 批处理过程

- 处理中：蓝 / cyan
- 成功：绿色
- 跳过：橙色
- 失败：低饱和红

原则：

- 使用 outline + 轻底色，不做厚重纯色块
- badge 是状态，不是主要按钮
- 历史页 badge 比扫描页再缩小一档

## 5. hover / glow 规则

### 行内光晕

- 使用整行连续 glow，不做按单元格分割
- 当前方向是鼠标跟随的 subtle glow
- 目的是强调“这一整行”，不是把某个单元格打亮

### 顶部 stats cards

- 是 console module，不是按钮
- hover 只允许轻微“通电响应”
- 不允许明显浮起或 dashboard 式点击感

### 复制反馈

- 复制成功提示使用局部 toast
- toast 锚定鼠标位置附近
- 不再占用页面顶部全局成功提示

## 6. integrated expanding badge 规则

状态操作区当前已经明确收敛到：

- badge 默认只显示状态
- hover 时向右自然延展
- badge 与操作区是一体，不是独立浮窗
- 只允许 hover 展开，不允许 click pin / click toggle
- 鼠标离开后延迟约 `300ms` 收起

当前动作区只显示真实存在的操作，不预渲染空槽位。后续扩展到 3~4 个操作时，也应继续沿用“动态扩展宽度”。

## 7. batch rail 规则

批处理反馈已经从普通 loading 升级为 batch rail：

- 位于表格上方
- 运行时自动展开
- 终态可折叠
- 显示状态、总数、成功/跳过/失败、进度条
- 不使用全屏遮罩

这个区域的定位是“系统状态条”，不是弹窗，也不是 SaaS 风格任务卡片。

## 8. 分页 / 排序 / 选择的风格要求

- 分页允许轻量 rail + 底部分页并存
- 排序不使用点击表头切换
- 排序控件应独立存在于 control rail
- 跨页选择状态应以内嵌状态条表达，不做大块批量操作条

## 9. 已明确不要回退的方案

以下方向已经明确不再建议回退：

- 传统 SaaS admin 表格风格
- 白底或高亮业务后台视觉
- 独立悬浮的大操作抽屉
- 状态 badge 作为 click-to-open 控件
- 复制后在页面顶部显示全局成功提示
- 历史页完全复用扫描页的大表格节奏
- 强调“所有东西都能点”的产品感

## 10. 后续新增能力时的 UI 原则

未来会继续引入刮削能力、更多勾选操作和批量动作。新增功能时优先遵循：

1. 维持 dark / console / devtool 主方向
2. 优先在现有 rail / row / badge 体系上扩展
3. 少做全新浮层，多做系统化的内嵌状态结构
4. 历史页优先做成“记录系统”，不要继续堆操作
