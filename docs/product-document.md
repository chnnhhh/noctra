> 更新时间：2026-03-27
>
> 参考文档：`docs/project-overview.md`
>
> 说明：本文基于当前仓库代码、前端交互、测试用例与运行脚本倒推出产品定义，描述的是“已经实现并可用”的产品能力，而不是未来规划。若文档与实现冲突，以 `app/`、`static/`、`tests/` 中的当前代码为准。

## 1. 产品定位

Noctra 是一个面向本地机器和 NAS 场景的 JAV 文件整理工作台。

它解决的不是“自动刮削一切”的问题，而是一个更具体的整理问题：

- 从混乱的下载目录中递归扫描视频文件
- 识别可用番号
- 预览整理后的目标路径
- 在用户确认后执行搬运
- 保留可回看的整理历史

产品核心强调 3 件事：

- 可见：整理前先看结果，不做黑盒式静默搬运
- 可控：只允许明确可整理的项进入执行集合
- 可追踪：整理后保留记录，避免重复处理同一番号

## 2. 目标用户与典型场景

### 目标用户

- 在本地硬盘或 NAS 上维护个人媒体目录的用户
- 希望统一 JAV 文件命名和目录结构的用户
- 不希望因为误识别、重复文件或覆盖风险而直接自动搬运的用户

### 典型场景

#### 场景 A：下载目录整理入库

用户把新下载的视频放进 `source` 目录，扫描后看到可识别番号和目标路径，确认无误后批量整理到 `dist`。

#### 场景 B：同番号出现多个候选文件

同一番号可能同时存在字幕版、无码版、不同扩展名或不同大小的文件。系统会自动选出一个“主候选”作为待处理，其余标记为重复，减少误操作。

#### 场景 C：目标目录里已经有同番号文件

如果目标文件已经存在，或历史上该番号已经被整理过，系统会标记为“已存在”，不重复覆盖。

#### 场景 D：用户只想清理 source

对于未识别、重复、已存在等条目，用户可以直接删除原始文件，或者仅忽略该扫描记录，让它后续不再出现。

#### 场景 E：回看最近整理结果

用户切到历史页，查看最近已经整理完成并且源文件已经离开 `source` 的记录。

## 3. 当前版本提供什么

### 已实现能力

- 递归扫描 `SOURCE_DIR`
- 自动跳过 `DIST_DIR`
- 支持视频扩展名过滤
- JAV 番号识别
- 目标路径实时预览
- 同番号候选自动判重与优先级排序
- 扫描页筛选、排序、分页、跨页勾选
- 单条整理与批量整理
- 批处理面板、进度条、成功/跳过/失败计数
- 批处理取消
- 历史记录页
- 删除原始文件
- 忽略扫描记录
- 本地运行、Docker 运行、NAS 部署

### 明确不在当前版本内

- 元数据刮削
- NFO / poster / fanart 生成
- 独立字幕文件联动处理
- 多层目录分类策略
- 自动重命名演员目录或主题目录
- 复杂规则引擎
- 多用户权限体系
- 自动后台监控目录并实时整理

## 4. 产品核心对象

### 目录语义

- `SOURCE_DIR`：待扫描的原始文件目录
- `DIST_DIR`：整理后的输出目录
- `DB_PATH`：SQLite 数据库路径

### 文件记录

系统围绕“扫描记录”管理每个文件，核心字段包括：

- 原始路径
- 识别到的番号
- 目标路径
- 当前状态
- 文件大小与修改时间
- 创建时间 / 更新时间

### 两个主要视图

- 扫描页：处理当前待整理的 source 文件
- 历史页：回看已真正整理完成的记录

### 顶部统计卡

顶部 5 张统计卡在产品中承担全局状态概览，包含：

- 总文件数
- 已识别
- 未识别
- 待处理
- 已处理

统计口径基于数据库中的非 `ignored` 记录，而不是当前分页结果。

## 5. 端到端产品流程

### 第一步：扫描目录

用户点击“扫描目录”后，系统会：

1. 递归遍历 `SOURCE_DIR`
2. 跳过 `DIST_DIR` 及其子目录
3. 只保留视频文件
4. 尝试从文件名中识别番号
5. 对已识别文件生成目标路径预览
6. 根据目标是否存在、是否与同番号文件冲突，计算状态
7. 把结果写入数据库并返回给前端

### 第二步：查看扫描结果

扫描页表格会展示：

- 番号
- 原始文件名
- 目标路径预览
- 当前状态

用户可以在这一层完成判断，而不是直接搬运。

### 第三步：筛选与选择

用户可以：

- 通过筛选器查看某一类状态
- 通过排序器调整顺序
- 使用分页浏览长列表
- 在多页之间累计选择文件

但当前只有 `pending` 状态允许加入批量整理集合。

### 第四步：执行整理

用户点击“执行整理”后：

- 前端会先弹出确认框
- 确认框展示本次将处理的番号集合
- 确认后创建一个批任务
- 批任务以串行方式逐个处理文件

### 第五步：查看批处理进度

批任务开始后，页面顶部会立即出现批处理面板：

- 显示任务状态
- 显示总数、成功、跳过、失败
- 显示进度条
- 表格行内状态同步刷新
- 支持取消当前批任务

### 第六步：同步扫描页

批任务完成后，前端会自动重新扫描一次，确保页面上的状态与文件系统真实状态一致。

### 第七步：回看历史

整理成功并且源文件已经不在原路径后，该记录会出现在历史页。

## 6. 页面与交互定义

### 6.1 扫描页

扫描页是当前产品的主工作台，包含：

- 顶部统计卡
- “扫描目录”按钮
- “执行整理”按钮
- 当前页选中 / 取消选中
- 筛选器
- 排序器
- 每页数量切换
- 分页器
- 扫描结果表格
- 批处理状态面板

#### 扫描页筛选项

- 全部
- 已识别
- 未识别
- 待处理
- 重复
- 已存在

#### 扫描页排序项

- 默认排序
- 按番号
- 按状态

#### 扫描页分页

- 默认每页 50
- 可切换为 20 / 50 / 100
- 支持跨页累计勾选
- 支持当前页全选与半选态

#### 扫描页行级动作

当前可操作状态与动作如下：

| 状态 | 可批量勾选 | 行级整理 | 删除 / 忽略 |
| --- | --- | --- | --- |
| `pending` | 是 | 是 | 是 |
| `duplicate` | 否 | 是 | 是 |
| `target_exists` | 否 | 否 | 是 |
| `skipped`（前端显示“未识别”） | 否 | 否 | 是 |
| `processed` | 否 | 否 | 否 |
| `failed` | 否 | 否 | 否 |

补充说明：

- `duplicate` 不能加入批量集合，但可以单独整理
- 点击番号本身也可以切换勾选状态，但仅限可选择项
- 点击源文件路径或目标路径会复制完整路径

### 6.2 批处理面板

批处理面板是扫描页里的常驻进度组件。

#### 面板特点

- 批任务创建后立即出现
- 默认展开显示详细信息
- 运行中持续可见
- 结束后保留结果，不自动消失
- 用户可手动收起
- 取消按钮仅在任务排队或运行中可用

#### 用户可见的批任务状态

- `queued`
- `running`
- `completed`
- `failed`
- `cancelled`

#### 用户可见的批任务子项状态

- `pending`
- `processing`
- `success`
- `skipped`
- `failed`

#### 当前交互细节

- 前端以 400ms 频率轮询批任务状态
- 面板最短显示时间为 800ms，避免闪现
- 运行期间禁止再次发起新的批处理

### 6.3 历史页

历史页用于回看整理结果，不再承担操作型工作流。

#### 当前行为

- 默认只展示真正已处理记录
- 默认按整理时间倒序
- 支持按时间、番号、状态排序
- 支持分页
- 不提供勾选和行级动作

#### 当前展示口径

只有同时满足以下条件的记录才会进入历史页：

- 数据库状态为 `processed`
- `original_path` 对应的源文件已经不存在

这意味着历史页展示的是“真正已经从 source 移走”的结果，而不是单纯写过一次状态。

## 7. 识别与整理规则

### 7.1 扫描范围规则

- 递归扫描 `SOURCE_DIR`
- 跳过 `DIST_DIR` 及其子目录
- 仅处理以下扩展名：
  - `.mp4`
  - `.mkv`
  - `.avi`
  - `.wmv`
  - `.mov`

### 7.2 番号识别规则

当前识别逻辑以“提取纯番号”为目标，特点如下：

- 返回统一大写的番号
- 第一段必须含字母，纯数字开头不识别
- 支持常见番号，也支持 `FC2-PPV-1234567` 这类形式
- 识别前会清理常见展示性标记

当前会被识别层去掉的内容包括：

- `-C`
- `C`
- `-UC`
- `UC`
- `字幕版`
- `字幕`
- `[Uncensored]`
- `Uncensored`

#### 识别示例

| 原文件名 | 识别结果 |
| --- | --- |
| `SSIS-123.mp4` | `SSIS-123` |
| `FPRE-123C.mp4` | `FPRE-123` |
| `MVSD-662-UC.mkv` | `MVSD-662` |
| `ABC-123_字幕版.mp4` | `ABC-123` |
| `ABC-123 [Uncensored].mp4` | `ABC-123` |
| `HMN-439-C.H265.mp4` | `HMN-439` |
| `CEMD-721ch.mp4` | `CEMD-721` |
| `123-456.mp4` | 不识别 |

### 7.3 目标目录与文件名规则

目标目录始终使用纯番号：

```text
/dist/{番号}/
```

目标文件名规则为：

```text
{番号}{规范化后缀}{扩展名}
```

#### 后缀规范化规则

- `C` / `-C` / `CH` / `ch` / `字幕` / `字幕版` -> `-C`
- `UC` / `-UC` / `Uncensored` -> `-UC`
- 没有明确语义标记 -> 不追加后缀

#### 文件名生成特点

生成逻辑不是“在原文件名上做局部修补”，而是基于：

- 标准番号
- 标准后缀
- 原始扩展名

重新组合目标文件名，因此会天然去掉番号后的演员名、剧情文本、编码标记等冗余内容。

#### 生成示例

| 原文件名 | 目标文件名 |
| --- | --- |
| `FPRE-123C.mp4` | `FPRE-123-C.mp4` |
| `FPRE-123_字幕版.mp4` | `FPRE-123-C.mp4` |
| `HMN-439-C.H265.mp4` | `HMN-439-C.mp4` |
| `CEMD-721ch.mp4` | `CEMD-721-C.mp4` |
| `ABC-123 [Uncensored].mp4` | `ABC-123-UC.mp4` |
| `MEYD-695 出轨xxx@北野未奈.mp4` | `MEYD-695.mp4` |

### 7.4 重复候选优先级规则

如果同一批扫描结果里出现多个同番号候选文件，系统会在 `pending` 候选中自动选出 1 个主候选，其余标记为 `duplicate`。

当前优先级规则如下：

1. 后缀类型优先级：`UC > C > SUB > PLAIN`
2. 同一后缀类型下，优先选择文件更大的候选
3. 若仍相同，按自然排序比较文件名
4. 若仍相同，按自然排序比较原始路径

这里的后缀类别含义：

- `UC`：无码 / uncensored
- `C`：明确的 `-C` / `C` / `CH`
- `SUB`：通过“字幕 / 字幕版”识别出来的字幕类
- `PLAIN`：普通无后缀候选

#### 重要产品行为

- 系统会推荐一个主候选，但不强制用户只能整理它
- `duplicate` 项不能批量勾选，但可以单条整理
- 一旦同番号中的任意一个候选整理成功，其余同番号记录会被更新为 `target_exists`

### 7.5 已存在判定规则

系统把以下情况统一视为“已存在”：

- 当前目标路径对应的文件已经存在
- 历史上已有同番号记录被成功整理，且源文件已经不在原路径

第二条意味着：即使新来的文件在后缀或扩展名上与历史记录不同，只要是同番号，当前产品也会优先阻止重复入库。

## 8. 状态体系

### 8.1 扫描记录状态

| 内部状态 | 前端显示 | 含义 |
| --- | --- | --- |
| `pending` | 待处理 | 可进入整理流程的主候选 |
| `duplicate` | 重复 | 同番号下的非主候选 |
| `target_exists` | 已存在 | 目标已存在或历史已整理过同番号 |
| `skipped` | 未识别 | 当前未识别到可用番号 |
| `processed` | 已处理 | 已整理完成 |
| `failed` | 失败 | 整理过程中失败的内部状态 |
| `ignored` | 不展示 | 用户选择忽略，后续扫描不再显示 |

补充说明：

- `ignored` 不进入扫描页，也不计入顶部统计
- `failed` 在当前产品里不是主工作流状态，批处理完成后前端会重新扫描，因此列表通常会回到重新计算后的真实状态

### 8.2 批任务状态

| 状态 | 含义 |
| --- | --- |
| `queued` | 已创建，等待开始 |
| `running` | 正在处理 |
| `completed` | 已结束，至少有部分项完成或跳过 |
| `failed` | 全部项失败 |
| `cancelled` | 用户请求取消后终止 |

### 8.3 批任务取消语义

- 取消是“请求取消”，不是回滚
- 已经完成的项会保留结果
- 尚未开始的项保持原有状态，不会被强制改写

## 9. 删除与忽略

扫描页中的可操作项支持两类清理动作：

### 删除原始文件

- 如果源文件存在：删除源文件，并删除数据库记录
- 如果源文件已不存在：直接删除数据库记录

适用场景：

- 已存在于目标目录，不想保留 source 副本
- 未识别文件判断为无价值，需要直接清理

### 仅删除扫描记录

本质上是把记录标记为 `ignored`：

- 源文件保留不动
- 后续扫描不再显示该记录

适用场景：

- 这个文件暂时不想处理
- 不希望它持续出现在工作台里

## 10. 历史记录口径

当前历史页不是完整审计日志，而是“整理完成列表”。

它有几个重要边界：

- 只显示 `processed`
- 只显示源文件已经离开的记录
- 不显示 `ignored`
- 不显示 `target_exists`
- 不显示 `duplicate`
- 不显示 `skipped`

因此它更适合用于“回看最近整理了什么”，而不是做全量问题排查。

## 11. 部署与运行形态

### 本地

仓库脚本默认以本地工作流运行，常见访问地址是：

```text
http://127.0.0.1:4020
```

### Docker / NAS

容器内的标准路径语义为：

- `/source`
- `/dist`
- `/app/data`

容器内部服务端口为 `8000`，常见做法是映射到宿主机的 `4020`。

### 健康检查

`GET /api/health` 当前会返回：

- profile
- source_dir
- dist_dir
- db_path
- cwd

它是排查“服务正在操作哪一套目录”的第一入口。

## 12. 当前产品约束与已知边界

### 产品边界

- 产品只负责基于文件名识别和整理，不负责富媒体信息补全
- 当前目录结构固定为“纯番号目录”
- 当前不支持覆盖已存在目标文件
- 当前没有批量重试入口
- 当前没有服务端持久化的批任务历史页

### 实现约束

- 批任务状态保存在进程内存中，服务重启后不会恢复
- 当前前端一次只围绕一个可见批任务工作，运行中会禁用再次发起
- 历史页的“整理时间”实际取自记录的 `updated_at`
- `force_rescan` 参数已经存在，但当前扫描行为本质上仍以实时全量扫描为主

## 13. 一句话总结

Noctra 当前已经是一个可用的“本地 / NAS JAV 文件整理工作台”：它能扫描、识别、判重、预览、批量整理、取消任务、回看历史，并通过“先看后搬”的产品模型，把误操作和重复入库风险压到较低水平。
